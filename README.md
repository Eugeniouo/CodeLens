# CodeLens — Умный поиск по кодовой базе

CodeLens - прототип RAG-системы для семантического поиска по исходному коду. Система позволяет разработчикам находить нужные фрагменты кода (функции и классы), формулируя запросы на естественном языке (русском или английском), даже если слова в запросе не совпадают с названиями переменных или функций.

---

## Архитектура системы

Проект состоит из четырех основных компонентов, полностью реализованных на Python 3.12:

1. **Модуль парсинга и векторизации (`parser.py`, `parser_java.py`, `embedder.py`):**
   * **Парсинг:** Используется встроенный модуль `ast` для Python и библиотека `tree-sitter` для Java. Код разбирается на абстрактные синтаксические деревья (AST), из которых извлекаются классы, методы и их docstring.
   * **Векторизация:** Модель `intfloat/multilingual-e5-small` преобразует текст и код в эмбеддинги. В тексты автоматически добавляются префиксы `passage:` и `query:` для обеспечения более высокого качества семантического поиска.
2. **Векторная база данных (ChromaDB):**
   * Локальное хранилище векторных представлений и метаданных (путь к файлу, тип сущности, имя функции/класса, номера строк, javadoc/docstring). Поиск выполняется по косинусному расстоянию.
3. **Веб-интерфейс (Streamlit):**
   * Полнофункциональное приложение для взаимодействия с системой без использования сторонних frontend-фреймворков.
4. **Аналитический LLM-модуль (`llm.py`):**
   * Интеграция с моделями через Groq API для генерации связных ответов и объяснения логики найденного кода.

## Обоснование стратегии чункования

**Мы используем семантическое разбиение на основе AST-деревьев (1 чанк = 1 функция или 1 класс).**

В отличие от «наивного» текстового разбиения (например, по фиксированному числу символов или токенов), подход на основе AST гарантирует, что логический контекст исполнения остается неразрывным.

**Почему это важно:**
1. **Сохранение семантики:** Функция или класс - это минимальная, атомарная и законченная единица бизнес-логики.
2. **Точность векторизации:** Сигнатура, docstring и тело функции попадают в один эмбеддинг, что повышает метрику Precision@5. Поисковая модель оценивает контекст целиком, а не обрывок кода.
3. **Читаемость результата:** Пользователь получает в интерфейсе готовый к анализу блок кода.

## Возможности веб-интерфейса Streamlit

Интерфейс разделен на три функциональных модуля:
* ** Семантический поиск:** Строка ввода запроса, настройка ответа LLM и выдача карточек с кодом. Система автоматически определяет язык файла (Python/Java) и применяет соответствующую подсветку синтаксиса.
* ** Индексация базы:** Модуль управления векторной базой данных через UI. Позволяет указать путь к новому репозиторию, выбрать режим полной очистки базы (Reset) и запустить синтаксический анализ и векторизацию в один клик.
* ** Валидация системы:** Страница автоматического тестирования поискового движка на валидационном датасете `eval_questions.json` с визуализацией метрик Precision@5, MRR и средней задержки (Latency).

## Запуск

### Docker Compose

1. (опционально) Для работы LLM выполните:
   ```
   cp .env.template.env
   ```
   и установите ключ API_KEY для API ключа groq в .env файл.

2. В корневую директорию проекта распакуйте кодовую базу или, при необходимости, замените точку монтирования в docker-compose.yml.

3. Запустите программу. Для выполнения необходим установленный Docker:
   ```bash
   docker compose up --build -d
   ```

4. Откройте в браузере: http://localhost:8501.

5. Проиндексируйте базу в интерфейсе системы (кнопка Индексация базы), указав полный путь или относительный внутри контейнера.

   Пример:
   ```
   /app/dataset_case3_v1/gymhero
   dataset_case3_v1/gymhero
   ```
   
## Локальный запуск

Для локального запуска также актуальны шаги 1. и 2. из секции "Docker Compose".

1. **Создание виртуального окружения и установка зависимостей**:
```bash
python -m venv .venv
source .venv/bin/activate
pip install .
```
Потребуется pip>=21.3.

Если установлен `uv`, используйте `uv sync`.

2. **Индексирование кодовой базы** (также доступно в веб-интерфейсе):
```bash
python index.py <path_to_codebase_dir>
```

3. **Запуск UI:**
```bash
streamlit run app.py
```

## Примеры запросов и ответов

Приведены урезанные ответы системы в режиме LLM.


### Пример 1

> **Запрос:** *«Как в проекте создаётся токен доступа и какой срок его жизни?»*

**Результат:**
Токен доступа создаётся с помощью функции `create_access_token` из файла `gymhero/gymhero/security.py`. Эта функция принимает два параметра: subject (субъект, для которого создаётся токен) и expires_delta (время жизни токена).

Срок жизни токена доступа определяется параметром expires_delta функции `create_access_token`. Если этот параметр не указан, срок жизни токена устанавливается в соответствии с настройкой ACCESS_TOKEN_EXPIRE_MINUTES из файла `settings`.


### Пример 2

> **Запрос:** *«Где в проекте проверяется, является ли пользователь суперпользователем?»*

**Результат:**
Проверка, является ли пользователь суперпользователем, осуществляется в следующих местах проекта:

* **Файл:** `gymhero/gymhero/crud/user.py`
* **Сущность:** `UserCRUDRepository.is_super_user`
* **Описание:** Метод is_super_user проверяет, является ли переданный пользователь суперпользователем, возвращая True, если пользователь является суперпользователем, и False в противном случае.


### Пример 3

> **Запрос:** *«Как в проекте хранится и проверяется пароль пользователя при входе?»*

**Результат:**
Пароль пользователя хранится в зашифрованном виде в базе данных. Это можно увидеть в файле `gymhero/gymhero/api/routes/auth.py`, сущность register, где пароль пользователя передается функции security.get_password_hash() для хеширования перед сохранением в базе данных.

Проверка пароля пользователя происходит в файле `gymhero/gymhero/crud/user.py`, сущность UserCRUDRepository.authenticate_user. Здесь пароль пользователя проверяется с помощью функции verify_password(), которая сравнивает введенный пароль с хешем пароля, сохраненным в базе данных.


### Пример 4

> **Запрос:** *«Как формируется строка подключения к PostgreSQL из настроек приложения?»*

**Результат:**
Строка подключения к PostgreSQL формируется из настроек приложения с помощью функции build_sqlalchemy_database_url_from_settings, расположенной в файле `gymhero/gymhero/database/session.py`, сущность build_sqlalchemy_database_url_from_settings.

Эта функция принимает на вход экземпляр класса Settings из файла `gymhero/gymhero/config.py`, сущность Settings, который содержит настройки подключения к базе данных PostgreSQL, такие как имя пользователя, пароль, хост, порт и название базы данных.


### Пример 5

> **Запрос:** *«Как базовый репозиторий реализует создание записи с привязкой к владельцу?»*

**Результат:**
Базовый репозиторий реализует создание записи с привязкой к владельцу через метод create_with_owner. Этот метод принимает три параметра: db (сессия базы данных), obj_create (данные для создания новой записи) и owner_id (идентификатор владельца).
